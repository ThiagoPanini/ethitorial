# Command: start

Use this command when the operator invokes `/solo-dev-assistant start` to bootstrap a new greenfield project.

`start` creates seed documentation so downstream skills can refine it. It does not try to finish product strategy, final roadmap structure, architecture, or implementation planning.

## Contract

`start` may write only these five files under the target project:

- `README.md`
- `AGENTS.md`
- `docs/VISION.md`
- `docs/ROADMAP.md`
- `docs/CONTEXT.md`

It does not create ADRs, issues, branches, product code, lessons, guides, runbooks, boards, PRDs, specs, remote resources, or private tracking state.

If any destination file already exists, stop and ask before rerunning with `--force`.

## Adaptive Interaction

Ask no more than three rounds.

### Round 1: required

Ask these three questions, preferably together:

1. Em uma frase, que problema esse projeto resolve?
2. Para quem?
3. Como saber que V1 deu certo?

If the operator is using English, ask the same questions in English:

1. In one sentence, what problem does this project solve?
2. Who is it for?
3. How will we know V1 succeeded?

After the answers, evaluate two things:

- Specificity: answers should contain concrete noun phrases, not only generic words such as "software", "system", "app", "platform", "developers", "users", "ready", "working", "sistema", "aplicativo", "plataforma", "desenvolvedores", "usuarios", "pronto", or "funcionando".
- Sufficiency for templates: the five docs can be generated without hiding missing information.

If the answers are sufficient, generate the docs.

### Round 2: conditional, max 2 questions

If one or two answers are shallow, ask focused questions for the weakest points only.

Examples:

- If audience is "developers": "What kind of developer: full-stack web, mobile, DevOps, ML, embedded, or another group?"
- If V1 success is "it works": "Concretely: which flows, numbers, active users, or validated environment prove that V1 succeeded?"
- If problem is "organize things": "Which things, in what context, and what pain shows up today?"

Do not ask more than two questions in this round.

### Round 3: rare, max 2 questions

Ask only if anti-scope or stack are still blocking obvious template slots:

- "What is this project explicitly NOT?"
- "Is the stack already decided? If yes, what is it; if not, we will leave it for a later grilling session."

If ambiguity remains after this round, generate the docs anyway with visible `_TODO_` slots.

## Generation

From the collected answers, run the bundled renderer from the target project root:

```bash
python3 <skill-dir>/scripts/start.py --target "$PWD" --answers-json <answers.json>
```

The JSON should use these keys:

```json
{
  "project_name": "project-name",
  "problem": "one-sentence problem",
  "audience": "target audience",
  "v1_success": "concrete V1 success criterion",
  "anti_scope": "what this project is not",
  "stack": "decided stack, or empty",
  "non_functionals": "relevant non-functional attributes, or empty",
  "language": "pt-BR or en, optional"
}
```

If a key is unknown, omit it. The renderer turns missing optional values into visible `_TODO_` markers rather than inventing content.

The renderer detects the dominant language from the answers. If the language is ambiguous, it defaults to pt-BR for this operator.

## Roadmap Seed Rule

The generated `docs/ROADMAP.md` is a seed, not the final roadmap. Its header must say that it was generated by `/solo-dev-assistant start`, exists only to orient the first sessions, and can be refined or replaced by specialized skills or workflows such as `grill-me`, `grill-with-docs`, `domain-model`, `to-prd`, `to-issues`, GSD, or Spec Kit.

## Final Output

After generation, show the generated file list and next steps in the detected language:

```markdown
✓ 5 documentos gerados:
  - README.md
  - AGENTS.md
  - docs/VISION.md
  - docs/ROADMAP.md
  - docs/CONTEXT.md (esqueleto)

Próximos passos sugeridos:
  1. Refinar visão, escopo e trade-offs → /grill-me
  2. Preencher glossário e invariantes → /grill-with-docs
  3. Decidir o próximo movimento do processo → /solo-dev-assistant cycle
  4. Quando começar a executar → /solo-dev-assistant briefing
```
